# Production-деплой для реальной клиники

Полное руководство по развёртыванию Clinic MIS в production-окружении для действующей клиники. Два сценария: **локальный сервер в клинике** (рекомендуемый) и **облачный хостинг**.

---

## ВАРИАНТ 1: Локальный сервер в клинике (РЕКОМЕНДУЕМЫЙ)

**Почему именно так:**
- Медицинские данные не покидают территорию клиники — соответствие законодательству РТ о персональных данных
- Нет абонентской платы за облако
- Не зависит от качества интернета для внутренней работы
- Полный контроль над данными
- Возможность бэкапа на физический носитель

### Аппаратные требования

#### Минимальная конфигурация (клиника до 5000 пациентов, 10 одновременных пользователей)

| Компонент | Требование |
|-----------|-----------|
| **Процессор** | Intel Core i5 / AMD Ryzen 5 (4+ ядра, 2.5+ GHz) |
| **Оперативная память** | 8 GB DDR4 |
| **Диск** | 256 GB SSD (лучше NVMe) |
| **Резервный диск** | 500 GB HDD для бэкапов |
| **Сетевая карта** | Gigabit Ethernet |
| **ИБП** | 650 VA минимум (защита от скачков) |

**Примерная стоимость комплекта:** 5 000 - 8 000 сомони единовременно

#### Средняя конфигурация (5-20 тыс. пациентов, 20-50 пользователей)

| Компонент | Требование |
|-----------|-----------|
| **Процессор** | Intel Core i7 / Xeon E-series (6-8 ядер) |
| **Оперативная память** | 16-32 GB DDR4 ECC |
| **Диск** | 512 GB NVMe SSD + 2 TB HDD |
| **RAID** | RAID 1 для данных (зеркалирование) |
| **ИБП** | 1000-1500 VA |

**Примерная стоимость:** 12 000 - 25 000 сомони

#### Большая клиника / сеть филиалов (50+ тыс. пациентов)

| Компонент | Требование |
|-----------|-----------|
| **Сервер** | Dell PowerEdge T340 / HP ProLiant ML110 |
| **Процессор** | Intel Xeon Silver (8-12 ядер) |
| **Оперативная память** | 32-64 GB DDR4 ECC |
| **Диск** | 2x 1TB NVMe RAID 1 + 4TB HDD |
| **ИБП** | 2000+ VA |
| **Резервный сервер** | Для отказоустойчивости |

**Примерная стоимость:** 40 000 - 80 000 сомони

### Операционная система

**Рекомендую:** Ubuntu Server 22.04 LTS или Debian 12

- Бесплатная
- Долгая поддержка (до 2027 года)
- Стабильная, проверенная
- Огромное сообщество и документация

### Пошаговая установка

#### 1. Подготовка сервера

```bash
# Обновление системы
sudo apt update && sudo apt upgrade -y

# Установка базовых пакетов
sudo apt install -y curl git nano htop net-tools ufw fail2ban

# Настройка межсетевого экрана
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp    # SSH
sudo ufw allow 80/tcp    # HTTP
sudo ufw allow 443/tcp   # HTTPS
sudo ufw enable

# Fail2Ban (защита SSH от брутфорса)
sudo systemctl enable fail2ban
sudo systemctl start fail2ban
```

#### 2. Установка Docker

```bash
# Скрипт установки с официального сайта
curl -fsSL https://get.docker.com | sudo bash

# Добавить текущего пользователя в группу docker
sudo usermod -aG docker $USER

# Перелогиниться или:
newgrp docker

# Проверить установку
docker --version
docker compose version
```

#### 3. Клонирование проекта

```bash
cd /opt
sudo mkdir clinic
sudo chown $USER:$USER clinic
cd clinic

git clone https://github.com/YOUR_USERNAME/clinic-mis.git .
```

#### 4. Настройка переменных окружения

```bash
cp .env.example .env
nano .env
```

Заполни:

```bash
# База данных
DB_NAME=clinic_db
DB_USERNAME=clinic_user
DB_PASSWORD=сгенерируй_сложный_пароль_32_символа
DB_HOST=postgres
DB_PORT=5432

# Redis
REDIS_HOST=redis
REDIS_PORT=6379

# JWT — сгенерируй командой: openssl rand -hex 32
JWT_SECRET=сгенерируй_64_символа
JWT_EXPIRES_IN=15m
JWT_REFRESH_SECRET=сгенерируй_другие_64_символа
JWT_REFRESH_EXPIRES_IN=7d

# Приложение
APP_PORT=3000
APP_ENV=production
FRONTEND_URL=https://clinic.your-domain.tj

# MinIO (встроенное S3-хранилище)
S3_ENDPOINT=minio
S3_PORT=9000
S3_ACCESS_KEY=minio_admin
S3_SECRET_KEY=сгенерируй_сложный_пароль
S3_BUCKET=clinic-files

# SMS (Babilon-M, MegaFon TJ, или другой)
SMS_API_KEY=получи_у_провайдера
SMS_SENDER=ClinicName

# Email
SMTP_HOST=smtp.yandex.ru
SMTP_PORT=587
SMTP_USER=clinic@your-domain.tj
SMTP_PASSWORD=пароль_от_почты

# Порты Nginx
APP_PORT=80
APP_SSL_PORT=443
```

#### 5. Запуск приложения

```bash
# Сборка и запуск всех сервисов
docker compose -f docker-compose.prod.yml up -d --build

# Проверка статуса
docker compose -f docker-compose.prod.yml ps

# Логи
docker compose -f docker-compose.prod.yml logs -f backend
```

#### 6. Первоначальное заполнение БД

```bash
# Войти в контейнер backend
docker compose -f docker-compose.prod.yml exec backend sh

# Внутри контейнера запустить seed
npm run seed

# Выйти
exit
```

#### 7. Настройка домена и SSL

Если у клиники есть домен (например, `clinic.example.tj`):

```bash
# Установить Certbot
sudo apt install -y certbot python3-certbot-nginx

# Получить SSL-сертификат (замени на реальный домен)
sudo certbot --nginx -d clinic.your-domain.tj

# Автоматическое обновление сертификата
sudo systemctl enable certbot.timer
```

Certbot автоматически обновит конфиг Nginx для HTTPS.

#### 8. Настройка резервного копирования

Создай скрипт `/opt/clinic/backup.sh`:

```bash
#!/bin/bash
BACKUP_DIR=/opt/backups/clinic
DATE=$(date +%Y%m%d_%H%M%S)
mkdir -p $BACKUP_DIR

# Бэкап PostgreSQL
docker compose -f /opt/clinic/docker-compose.prod.yml exec -T postgres \
  pg_dump -U clinic_user clinic_db | gzip > $BACKUP_DIR/db_$DATE.sql.gz

# Бэкап файлов MinIO
docker compose -f /opt/clinic/docker-compose.prod.yml exec -T minio \
  tar -czf - /data | cat > $BACKUP_DIR/files_$DATE.tar.gz

# Удалить бэкапы старше 30 дней
find $BACKUP_DIR -type f -mtime +30 -delete

echo "Бэкап завершён: $DATE"
```

Сделай исполняемым и добавь в cron:

```bash
chmod +x /opt/clinic/backup.sh

# Редактировать cron
crontab -e

# Добавить строку: ежедневный бэкап в 2:00 ночи
0 2 * * * /opt/clinic/backup.sh >> /var/log/clinic-backup.log 2>&1
```

**Важно:** также копируй бэкапы на **внешний диск или облако** (Yandex.Disk, Google Drive) раз в неделю.

## ВАРИАНТ 2: Облачный хостинг

Если клиника не хочет держать физический сервер или нет IT-специалиста.

### Рекомендуемые провайдеры для Таджикистана

#### Для соответствия законодательству РТ
Медицинские данные граждан РТ должны храниться на серверах в РФ, Казахстане, Беларуси или внутри Таджикистана. **Не используй** европейские/американские хостинги для реальных медданных.

| Провайдер | Страна | Мин. конфигурация | Цена (примерно) |
|-----------|--------|---------------------|-----------------|
| **Yandex Cloud** | Россия | 2 vCPU, 4 GB RAM, 40 GB SSD | 300-500 сомони/мес |
| **Selectel** | Россия | 2 vCPU, 4 GB RAM, 40 GB SSD | 250-400 сомони/мес |
| **VK Cloud (Mail.ru)** | Россия | 2 vCPU, 4 GB RAM, 40 GB SSD | 300-500 сомони/мес |
| **Cloud4Y** | Россия | 2 vCPU, 4 GB RAM, 40 GB SSD | 300-450 сомони/мес |
| **PS.kz** | Казахстан | 2 vCPU, 4 GB RAM, 40 GB SSD | 250-400 сомони/мес |

### Процедура облачного деплоя

Шаги те же, что и для локального сервера (пункты 1-8 выше), только:
1. Заказываешь VPS у провайдера
2. Получаешь IP-адрес и SSH-доступ
3. Подключаешься по SSH: `ssh root@IP_АДРЕС`
4. Выполняешь установку по шагам 1-8 из раздела "Локальный сервер"

## Безопасность в production

### Обязательные меры

1. **Сильные пароли** — минимум 16 символов для сервера, 12 для пользователей (уже включено в систему)
2. **Двухфакторная авторизация** — для владельца и сисадмина (встроено в систему)
3. **Автоматические бэкапы** — ежедневные, хранение 30+ дней
4. **Внешние копии бэкапов** — на диск вне клиники или в облако
5. **Обновления ОС** — регулярные: `sudo apt update && sudo apt upgrade`
6. **Межсетевой экран** — UFW настроен, открыты только 22, 80, 443 порты
7. **Fail2Ban** — защита от брутфорса SSH
8. **Нерутовый SSH** — отключить вход под root, использовать sudo-пользователя
9. **SSH по ключу** — отключить вход по паролю
10. **HTTPS везде** — Let's Encrypt автоматически
11. **Мониторинг логов** — еженедельная проверка audit_logs в системе
12. **Физическая защита сервера** — в закрытом помещении с ограниченным доступом

### Соответствие законодательству РТ

- Закон РТ **"О защите персональных данных"** (от 03.08.2018, №1537)
- Закон РТ **"Об информации"**
- Закон РТ **"Об охране здоровья населения"**

**Требования:**
1. Данные хранятся на территории РТ или в странах с международными соглашениями (РФ, Казахстан)
2. Согласие пациента на обработку данных (встроено в систему — модуль Consent)
3. Журнал доступа к данным (встроено — Audit Log)
4. Защита от несанкционированного доступа (встроено — роли + 2FA)
5. Резервное копирование
6. Возможность удаления данных по запросу (soft delete уже есть)
